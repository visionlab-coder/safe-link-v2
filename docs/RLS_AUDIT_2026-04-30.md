# SAFE-LINK V2 RLS Audit

Date: `2026-04-30`

## Summary

PoC 기준으로는 동작하지만, 상용화 직전 기준으로 보면 핵심 테이블의 RLS가 아직 일관되게 굳어 있지 않다.

가장 큰 문제는 두 가지다.

1. `profiles`, `tbm_notices`, `tbm_ack`에 대해 저장소 기준의 핵심 정책 파일이 부족하다.
2. 일부 기존 정책은 현재 role 체계와 어긋난다.

## Concrete Findings

### 1. `messages`만 상대적으로 정리되어 있음
- 기준 파일:
  [supabase/FINAL_FIX_ALL.sql](/C:/Users/seowo/Documents/dev/seowon-projects/SAFE-LINK-V2/supabase/FINAL_FIX_ALL.sql:1)
- 현재 상태:
  `SELECT`, `INSERT` 정책은 있다.
- 남은 보강:
  수신자가 `is_read`를 업데이트할 수 있는 `UPDATE` 정책이 필요하다.

### 2. `profiles` 핵심 정책 부재
- 앱 요구:
  - 사용자는 자기 profile을 읽고 수정
  - `HQ_ADMIN`, `SAFETY_OFFICER`는 같은 `site_id` 사용자 목록 조회
  - `ROOT`, `HQ_OFFICER`는 전체 조회
- 현재 저장소 기준:
  위 계약이 명시된 RLS SQL이 없다.

### 3. `tbm_notices` 핵심 정책 부재
- 앱 요구:
  - 관리자/안전관리자는 자기 현장 TBM 생성
  - 근로자는 자기 현장 TBM 조회
  - 시스템 권한은 전체 조회
- 현재 저장소 기준:
  위 계약이 명시된 RLS SQL이 없다.

### 4. `tbm_ack` 핵심 정책 부재
- 앱 요구:
  - 근로자는 자기 서명만 insert
  - 관리자는 자기 현장 서명 현황 조회
  - 시스템 권한은 전체 조회
- 현재 저장소 기준:
  위 계약이 명시된 RLS SQL이 없다.

### 5. `construction_glossary` 정책이 예전 role 문자열 사용
- 기준 파일:
  [supabase/migrations/20260222_construction_glossary.sql](/C:/Users/seowo/Documents/dev/seowon-projects/SAFE-LINK-V2/supabase/migrations/20260222_construction_glossary.sql:38)
- 문제:
  `role = 'ADMIN'`을 기준으로 관리 정책이 작성돼 있다.
- 현재 앱 role:
  `HQ_ADMIN`, `SAFETY_OFFICER`, `ROOT`, `HQ_OFFICER`, `WORKER`
- 영향:
  glossary 관리 기능이 실제 관리자 role과 어긋날 수 있다.

### 6. `safety_education_library` 정책이 잘못된 컬럼 사용
- 기준 파일:
  [supabase/migrations/20260409_safety_education_library.sql](/C:/Users/seowo/Documents/dev/seowon-projects/SAFE-LINK-V2/supabase/migrations/20260409_safety_education_library.sql:27)
- 문제:
  `profiles.system_role in ('HQ_ADMIN', 'SAFETY_OFFICER')`를 사용한다.
- 실제 데이터 모델:
  `HQ_ADMIN`, `SAFETY_OFFICER`는 `role` 컬럼 값이지 `system_role` 값이 아니다.
- 영향:
  관리자 insert/update 정책이 의도대로 동작하지 않을 수 있다.

### 7. `live_translations` 읽기 범위가 넓음
- 기준 파일:
  [supabase/migrations/20260416_live_translations.sql](/C:/Users/seowo/Documents/dev/seowon-projects/SAFE-LINK-V2/supabase/migrations/20260416_live_translations.sql:22)
- 문제:
  `authenticated` 전체 읽기 허용
- 상용화 기준:
  현장 단위 `site_id` 기준으로 좁히는 것이 더 안전하다.

## Recommended Direction

1. helper function 기반으로 현재 사용자 `role`, `site_id`를 읽는 security definer 함수 도입
2. `profiles`, `tbm_notices`, `tbm_ack`, `messages` 핵심 정책 재정의
3. `construction_glossary`, `safety_education_library`, `live_translations` 정책을 현재 role 체계에 맞게 수정
